## 1. Overview

Elemental Design uses a two-layer `.md` file system: **spec files** written during development, and **component docs** generated by the app at runtime for its users.

**Design principle: Progressive disclosure applies to documentation too.** Every `.md` file contains only what is needed at its level. Depth lives in linked files, not in the same document. A reader — human or AI — should never have to read more than they need for the task at hand.

This system is not incidental. The way Elemental Design documents itself mirrors the way Elemental Design works. The `router→` pattern in docs is the same philosophy as `>>>` in the UI.

## 2. The router→ Convention

`router→` is a navigation primitive. It appears inline in a document at the moment a topic arises that has its own dedicated file.

### Syntax

```
router→ filename.md — one-line description of what that file contains
```

### Rules

- **Always inline.** Place `router→` at the point in the document where the topic becomes relevant. Never collect routers at the bottom of a file.
- **Always include a description.** The description is a one-line contract — it tells the reader whether they need to follow the link right now. Vague descriptions like "more info here" are not allowed.
- **Filename only, no path prefix** for files in the same directory. Use relative paths for files in subdirectories: `router→ subdir/Spec_X.md — ...`
- **One router per topic per file.** Do not link to the same file twice in one document.
- **`router→` is a reserved token.** It will not be confused with prose arrows (`→`), list markers, or flow diagrams. Grep for `router→` to get a full map of the document graph.

### What router→ is not

- Not a footnote. It appears in the flow of the document, not appended to it.
- Not a replacement for content. If something can be said in two sentences, say it — don't route to a new file for it.
- Not optional. If a system has a spec file, it must be linked from every file that touches that system.

## 3. Spec Files

### Purpose
Spec files define how Elemental Design is built. They are written collaboratively between Charlie and Claude (claude.ai) during the design phase, then read by Claude Code during implementation.

### Location
All spec files live in `specs/` at the project root.

### Naming
```
Spec_[SystemName].md
```
PascalCase system name. No spaces. Examples:
- `Spec_Concept_Elemental.md`
- `Spec_ObjectModel.md`
- `Spec_ProgressiveDisclosure.md`

### Structure
Every spec file follows this order. Omit sections that don't apply.

```
# Elemental Design — [System Name] Specification
**[Subtitle]**
Version | Date | Author

## Version History
## 1. Overview         ← what it is, design principle, scope boundary
## 2–N. [System sections]
## N+1. What This Is Not   ← explicit scope boundary
## N+2. Related Specs      ← router→ links to adjacent systems
```

### The Overview section must contain
1. What this system is — one sentence
2. The design principle — the philosophical guardrail for this system
3. Explicit scope boundary — what this system does NOT do

### Writing rules
- Say why, not just what. Every significant decision includes a rationale.
- Bold key terms on first introduction only.
- No marketing language. This is a design document.
- All constraints explicit. If a value is clamped, state the clamp. If an operation is ordered, state the order.

## 4. Component Doc Files

Component docs are a **product feature**, not dev documentation. Elemental Design generates them automatically when a user creates a Component. They are the component's living record — readable by AI agents, developers, and Elemental Design itself.

### Location
Generated alongside the project file the user is working in. Path is user-controlled.

### Naming
```
[ComponentName].component.md
```
Kebab-case component name. Examples:
- `game-tile.component.md`
- `nav-header.component.md`

### Structure
```
# [Component Name]

**Type:** Component — [Subtype]
**Created:** [date]
**Last modified:** [date]

## Structure
[size, layout parameters]

## Content
[color, text, image parameters]

## Behavior
[interaction, animation, state parameters]

## Notes
[freeform — never overwritten by the system]
```

### Generation rules
- Generated fresh from component state on every save.
- The **Notes section is never overwritten.** It is the one place a human can write freely without the system touching it.
- AI-authored parameter values are marked with `✦` inline.
- `router→` may appear in component docs to link to shared style definitions or design tokens.

router→ Spec_ComponentDocs.md — full component doc schema, generation logic, write-back rules

## 5. Document Graph

Every spec file must be reachable from `CLAUDE.md` or `Spec_Concept_Elemental.md` via `router→` links. Orphaned spec files — files with no inbound links — are considered incomplete.

To audit the document graph:
```bash
grep -r "router→" specs/
```

This returns every link in the system. Every spec filename should appear as a destination at least once.

---

## 6. What This System Is Not

- Not a wiki. There is no search, no tags, no backlinks. Navigation is explicit via `router→`.
- Not auto-generated. Spec files are written intentionally. They are not scraped from code.
- Not versioned per-commit. Version history is maintained manually in each file's Version History table.
