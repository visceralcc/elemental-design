# Elemental Design — Claude Code Entry Point

You are working on **Elemental Design**, a native macOS and iOS design tool built in SwiftUI.
Read this file first. It tells you how this project is organized and how to work in it.

---

## What Elemental Design Is

A native design tool that rejects the toolbar-and-panel metaphor. Users declare intent.
Elemental handles defaults. Complexity is revealed progressively, never shown upfront.

router→ specs/Spec_Concept_Elemental.md — full concept, philosophy, and interaction model

## Before You Write Any Code

1. **Find the relevant spec.** Every system has a spec file in `specs/`. Read it before touching anything.
2. **Follow the router links.** If a spec points to another with `router→`, follow it if your task touches that system.
3. **Do not invent conventions.** Naming, structure, and patterns are defined in the specs. Match them exactly.
4. **If no spec exists for what you're building, stop and say so.** Do not proceed without one.

router→ specs/Spec_DocumentSystem.md — how specs are written, the router→ convention, file naming rules

## The Two .md Systems

This project uses `.md` files in two distinct ways. Do not confuse them.

**1. Spec files** (`specs/`) — written by humans and AI during development.
Define how Elemental Design is built. You read these. You do not generate them.

**2. Component docs** (`*.component.md`) — generated by Elemental Design at runtime for its users.
Each Component a user creates gets its own `.md` file. These are a product feature, not dev docs.

router→ specs/Spec_ComponentDocs.md — component .md file structure, generation rules, router→ usage within them
